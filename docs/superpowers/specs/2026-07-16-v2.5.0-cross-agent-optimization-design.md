# v2.5.0 — cross-agent generalization, gate parity, token/limit economics

**Date:** 2026-07-16
**Status:** approved scope, spec for implementation planning
**Trigger:** external field feedback (5-point review) + owner's cross-agent and
token-optimization requirements.

## Goal

Close the verified gaps from the field feedback and generalize the template
beyond Claude Code: honest positioning, Go race detection, a cross-profile
parity contract, an agent-selection interview step with per-agent conditional
emission, and a token/session-limit economics layer that shifts heavy work to
secondary agents (Codex, Antigravity) when they are configured.

## Verified facts this spec relies on

- `profiles/go/scripts/qa.sh` runs `go test ./...` with no `-race`; no `-race`
  anywhere in the repo. Rust/Python/TS have no equivalent gap.
- README line 5 promises "a product manager, a designer, or a first-week
  coder"; the machine requires senior-dev discipline. Copy-only fix chosen.
- Supply-chain pinning is ALREADY DONE (v2.4.0): all GitHub Actions SHA-pinned
  with a CI check job, devcontainer images digest-pinned, uv installer
  sha256-verified, `degit@2.8.4` and `@upstash/context7-mcp@3.2.3` pinned.
  `docs/ROADMAP.md` still claims this is future work — stale text, not a gap.
- TypeScript enforces the 200-line cap via ESLint `max-lines`, not
  `linecap.sh` — parity holds, mechanism differs.
- All five subagents pin `model: sonnet`; `<token-discipline>` already has the
  model-economics rule; there is no cheap-tier utility agent.
- Claude Code facts (docs-verified): hooks CANNOT reroute a call to a cheaper
  model; the only supported cheap-model routing is a subagent with `model:`
  frontmatter. There is no API to query remaining session quota; limit
  awareness must be rule-based (react to harness limit warnings).

### 4. Agent selection (new interview question B13) — availability-aware, mutable

Design principle: the agent roster is NOT frozen at interview time. Someone
may install Codex/Antigravity/Gemini mid-project; the project must pick that
up without re-rendering. Selection state therefore lives in a runtime config
file, and only what genuinely cannot exist for a non-selected agent (the
`.claude/` directory) is a render-time conditional.

- **B13** (after B12): "Which agentic coders will drive this project?"
  Multi-select: `claude-code` (default), `codex`, `antigravity`, `cursor`,
  `other`. At least one required. **Availability probe:** before presenting
  the list, the interviewing agent probes the environment (`command -v codex`,
  `gemini`, etc., plus configured MCP servers) and marks each option
  installed / not installed. Not-installed agents may still be selected
  (recorded as `"planned"`); the default selection is what is available.
- **Runtime config `docs/agents.json`** (new always-on file, rendered from
  B13): machine-readable roster mapping each enabled agent to its offload
  roles — `utility` (cheap-tier mechanical work), `second_opinion` (review),
  `heavy_batch` (workload shifting near session limits) — plus its
  installed/planned status. The AGENTS.md workload-shifting and
  token-discipline rules reference THIS file at runtime, so a roster change is
  an edit to one JSON file, never a re-render. This file is the generalized,
  agent-agnostic form of an "offload config" — any agent the user has access
  to slots into a role; nothing is vendor-specific.
- **Mid-project re-selection: `/select-agents` skill.** A small skill shipped
  in generated projects (`templates/core/.claude/skills/select-agents/SKILL.md`)
  that re-runs the availability probe, presents the same selection list as
  B13, and rewrites `docs/agents.json` + `docs/agents.md`. Gated on
  `claude-code` like the rest of `.claude/`; for non-Claude agents,
  `docs/agents.md` documents the manual-edit path (the JSON is
  human-editable by design).
- **Interplay with B12:** B12 (Codex second-opinion reviewer) is asked only
  when `codex` is selected in B13. Validation fails closed if
  `codex_reviewer: yes` while `codex` is absent from `agents`.
  (Interview order note: B13 is asked before B12 in the flow, or B12's gate is
  re-checked — implementation plan decides; numbering stays as-is to avoid
  renumbering churn.)
- **Answers schema:** new required key `agents` in `docs/_init-answers.json`
  — a list of `{name: <enum>, status: installed|planned}` objects (the probe
  result is recorded at interview time so `render.py` stays deterministic:
  same answers, same bytes, including `docs/agents.json`). `render.py`
  validates names against the enum and fails closed on unknown values or an
  empty list.
- **Conditional emission (render.py rules):**
  - `.claude/agents/`, `.claude/hooks/`, `.claude/settings.json`,
    `.claude/skills/` (incl. `select-agents`) and the `CLAUDE.md` symlink are
    emitted only when `claude-code` is in `agents`.
  - `.claude/.template-version` stamp is ALWAYS emitted (upgrade-project
    depends on it) regardless of agent selection.
  - When `claude-code` is absent, `docs/agents.md` (below) must state which
    enforcement pieces were omitted and what replaces them (nothing, today —
    honest statement).
  - **Phase 5 gating:** the SKILL.md Phase 5 verification currently hard-codes
    `test -d .claude/agents`, `.claude/settings.json`, hook checks — these
    become conditional on `claude-code` being selected; `docs/agents.md` +
    `docs/agents.json` join the unconditional checks.
  - **Bootstrap remnants:** `install.sh` installs `.claude/skills/` (the
    init-project skill and required skills) into the target BEFORE the
    interview, so deselecting `claude-code` cannot un-create them. When
    `claude-code` is absent, the skill's final report tells the user those
    bootstrap-installed `.claude/` files are inert and safe to delete.
- **New always-on file `docs/agents.md`:** human-readable capability matrix
  rendered from per-agent snippet files in
  `templates/conditional/agents/<agent>.md` via a new `{{AGENT_MATRIX}}`
  placeholder. Each snippet states: what the agent reads (AGENTS.md), what
  enforcement it gets (subagents/hooks/MCP for Claude Code; rules-only for
  others today), its offload roles, and how to change the roster later
  (`/select-agents`, or edit `docs/agents.json` directly). Registered in the
  placeholder table, Phase 4 behavior table, Phase 5 core-file check, and
  `upgrade-project/SKILL.md` Phase 3-A (both `agents.md` and `agents.json`;
  an old project missing them is reported with a pointer to
  `/select-agents`).
- **Golden fixtures:** add the `agents` list to all six existing fixtures
  (claude-code installed; plus codex where the fixture already opts into the
  Codex reviewer); add one new fixture with `claude-code` ABSENT (e.g.
  codex-only TypeScript) proving `.claude/` agents/hooks/settings/skills and
  the symlink are skipped while the version stamp, `docs/agents.md`, and

### 5. Token & session-limit economics

- **New `@utility` subagent** `templates/core/.claude/agents/utility.md`,
  `model: haiku`, restricted to mechanical chores: git housekeeping, log
  mining/filtering, bulk renames, doc formatting, status summaries. Explicitly
  forbidden from writing product code or tests. Ships inside `.claude/agents/`
  so it is automatically gated on the `claude-code` selection.
- **`<token-discipline>` clause (FW-BLOCK version bump to v2.5.0):** delegate
  multi-step mechanical work to the cheapest configured agent (`@utility` on
  Claude Code); never burn default/strong-tier context on it.
- **Workload-shifting rule** (in the same block + expanded in
  `docs/agents.md`): when session-limit warnings appear, or a task is heavy
  batch work (large migrations, broad audits, long test-fix loops), consult
  `docs/agents.json` and dispatch the heavy work to an enabled
  `heavy_batch`-role agent (Codex CLI via `codex exec`, Antigravity), keeping
  Claude Code as orchestrator/reviewer — shifting progressively more work to
  the secondary agent as limit pressure grows. The rule reads the runtime
  config, not baked-in agent names, so a roster change applies immediately.
  State honestly that there is no quota API; triggers are harness warnings
  and the user's own `/status` judgment.
- **Subagent roster note** in core AGENTS.md (the "Picking a model per call"
  section) gains `@utility` with its haiku default.

## Error handling

- `render.py` fails closed on: empty/unknown `agents` values, `codex_reviewer:
  yes` without `codex` in `agents`, any unresolved `{{AGENT_MATRIX}}`.
- The parity check fails CI on a missing capability row or missing evidence.
- Old answers files without `agents` fail validation (schema change is part of
  the release; fixtures all updated). `upgrade-project` treats a project
  without `docs/agents.md`/`docs/agents.json` as "pre-2.5.0" and reports it
  with a pointer to `/select-agents` — it cannot infer the agent roster.
- `/select-agents` never edits anything except `docs/agents.json` and
  `docs/agents.md`; it fails closed (no partial write) if either file is
  malformed.
